# Contributing — Public-Safe Feedback Boundaries

> This repository accepts limited public-safe documentation feedback.  
> Contribution does not create ownership, control, or entitlement over Model 1 v1.0.

## 1. Purpose

This file defines how outside readers may offer feedback without expanding or weakening the public v1.0 boundary.

Model 1 v1.0 is a narrow first-person stop-loss guardrail protocol. It is not an open-ended community framework, governance platform, therapeutic method, agent system, or compliance product.

Public contribution is limited to documentation clarity and public-safe interface improvement.

## 2. Core rule

```text
Public availability does not create entitlement.
Suggestions do not create obligation.
Contribution does not create ownership.
```

The maintainer may decline, close, ignore, or defer any contribution that expands scope, weakens safety, requests private material, or attempts to change the core model.

## 3. Contributions that may be considered

Public-safe contributions may include:

- typo fixes
- broken link fixes
- formatting or readability improvements
- clearer wording that preserves the existing meaning
- public-safe terminology suggestions
- documentation consistency fixes
- non-identifying synthetic examples
- translation suggestions that preserve scope and safety boundaries
- clarification requests about public documents

These contributions must not require private context to evaluate.

## 4. Contributions that are out of scope

The following are not accepted:

- core model rewrites
- requests to publish private case chains
- requests for runtime thresholds, observer packs, or calibration logic
- proposals to turn Model 1 into an AI governance platform
- proposals to turn Model 1 into a GRC control catalog
- proposals to add ISO / NIST / EU AI Act compliance mappings as core material
- agent execution implementations
- autonomous workflow implementations
- risk scoring systems
- profiling, ranking, diagnosing, or judging people
- relationship analysis systems
- medical, legal, emergency, HR, employment, or safety triage replacements
- persuasion, influence, manipulation, pressure, or negotiation tactics
- commercial repackaging or training material without prior written permission

## 5. Preferred workflow

The repository uses a lightweight review gate.

Direct maintainer commits may be used for:

- obvious typo fixes
- broken links
- formatting corrections
- non-structural wording cleanup

Pull requests should be used for:

- new public files
- README map changes
- license or commercial-use changes
- contribution rule changes
- public examples or trace examples
- methodology or teaching-material additions
- any change that may alter public interpretation of Model 1 v1.0

External contributions must use pull requests.

A pull request is not merely a coding convention. In this repository, it is a public review gate:

```text
proposal → diff → review → merge / reject
```

## 6. Pull request guidance

A pull request should be small, specific, and public-safe.

A good pull request:

```text
- fixes a typo
- clarifies a sentence without changing scope
- improves a table label
- corrects a broken link
- proposes a synthetic example with no real-person trace
```

A pull request should not:

```text
- change the core judgment logic
- add private cases
- add executable code that acts on signals
- convert signals into enforcement actions
- make Model 1 look like a compliance certification system
- expand the repo into a governance platform
```

## 7. Issues and discussion guidance

Issues may be opened for:

- documentation ambiguity
- unclear public terminology
- broken links
- boundary clarification
- public-safe wording suggestions

Issues should not be opened for:

- personal advice
- third-person analysis
- crisis or emergency support
- medical, legal, financial, employment, or safety decisions
- requests to analyze a specific person, relationship, company, or case
- attempts to use the model to influence or pressure others

## 8. Synthetic examples only

If proposing examples, use synthetic or fully de-identified scenarios.

Do not submit:

- real medical cases
- real relationship cases
- real workplace disputes
- identifiable people, companies, locations, dates, screenshots, or private facts
- private messages or personal documents

A safe example should be reduced to structure only:

```text
A workflow proposes a consequential action.
Rollback is unclear.
A human owner has not reviewed the risk.
Expected posture: PAUSE or ASK_HUMAN.
```

## 9. No entitlement principle

Submitting feedback does not entitle a contributor to:

- merge rights
- roadmap influence
- model ownership
- authorship credit
- governance participation
- private context access
- explanation of private rejection reasons
- future commercial rights

The maintainer may preserve silence or close a contribution when engaging would create scope drift or safety risk.

## 10. Commercial boundary

Commercial use, commercial redistribution, product integration, training use, advisory use, compliance use, risk consulting use, or enterprise workflow use requires prior written permission.

See [`COPYRIGHT_AND_LICENSE.md`](COPYRIGHT_AND_LICENSE.md).

## 11. Safety boundary

All contributions must preserve the public v1.0 boundaries:

```text
First-person only.
No profiling.
No manipulation.
No automated authority.
No professional substitute.
No private case exposure.
Human Final Gate remains mandatory.
```

## 12. Maintainer discretion

The maintainer may reject contributions for reasons including but not limited to:

- scope expansion
- safety weakening
- ambiguity
- private-context dependency
- commercialization risk
- misuse risk
- overfitting to adjacent AI governance or GRC frameworks
- attempts to convert the model into executable authority

No public explanation is required for every rejection.

## 13. One-line summary

```text
You may help clarify the public scaffold, but you may not claim, steer, expand, operationalize, or own the model.
```
